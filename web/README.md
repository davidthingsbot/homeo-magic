# Homeo-Magic Web UI

A web-based repertorization tool for homeopathy. Find remedies that match multiple rubrics, ranked by total score.

## Features

- **Rubric Search**: Fuzzy, typo-tolerant search across 74,000+ rubrics with multi-token matching
- **Session Memory**: Recently selected rubrics appear first in suggestions
- **Intersection Matching**: Only shows remedies present in ALL selected rubrics
- **Score Ranking**: Remedies ranked by total score (sum of weights 1-3)
- **Score Breakdown**: Expandable view showing contribution per rubric
- **Kent's Materia Medica**: Inline lecture panel with passage highlighting for selected rubrics
- **Lazy Loading**: Split data files loaded on demand for fast initial page load

## Tech Stack

- Next.js 16 (App Router, static export)
- React 19
- Tailwind CSS 4
- TypeScript

## Getting Started

```bash
# Install dependencies
npm install

# Copy split data files to public/data/
npm run setup

# Run development server
npm run dev
```

## Building & Serving

### For GitHub Pages (production default)

```bash
npm run build
```

This bakes in `basePath: "/homeo-magic"` so all asset paths are prefixed with `/homeo-magic/`, matching the GitHub Pages deployment URL.

### For local / Tailscale Funnel serving

```bash
SKIP_BASE_PATH=1 npx next build
cp -r public/data out/data
npx serve out -l 10101
```

`SKIP_BASE_PATH=1` builds without the `/homeo-magic/` prefix, so assets resolve at root `/`. You must also copy the split data files into `out/data/` after building, since the static export doesn't include them.

### Important: Always copy data after build

The `out/` directory is regenerated on every build. The split data files in `public/data/` are NOT automatically copied to `out/data/`. Always run:

```bash
cp -r public/data out/data
```

after `npm run build` or `npx next build` before serving the `out/` directory.

## Data Files

The app lazy-loads split data from `public/data/`:

- `symptoms/` - Rubrics split by body system/subcategory (~2-3K files)
- `remedies/` - Remedies split by first letter
- `kent/` - Kent's Materia Medica (profiles, passage index, per-remedy markdown)
- `symptom_pairs.json` - Pre-computed rubric pair data
- `default-symptoms.json` - Default/common rubrics for quick access

These are generated by the Python data pipeline in `../data/` and copied by `npm run setup`.

### TDD Workflow

We follow test-driven development. When making changes:

1. Write/update tests first for the expected behavior
2. Run tests -- confirm they fail
3. Write the minimum code to make tests pass
4. Run full test suite to catch regressions

## Algorithm

The repertorization algorithm:

1. For each selected rubric, gather all remedies and their weights
2. Filter to only remedies that appear in **all** selected rubrics (intersection)
3. Sum the weights for each remedy across rubrics
4. Sort by total score descending

This ensures results are highly specific -- a remedy must address every selected rubric to appear.
